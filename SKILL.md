# Story Studio — 多Agent小说创作剧组 v2.0

## Overview

Story Studio 是一个模仿影视剧组工作机制的多Agent小说创作系统。

**核心分界**：Skill = 引擎（不含任何故事内容）。Project = 世界实例（每个故事独立管理）。Agent启动时读取项目文件，所有示例用 `{占位符}`。

v2.0 在 v1.0 实验基础上全面重写：入场关卡（现场监理）、锚点约束系统、Scene Spec 驱动、5角色评审质控、统筹动态编排、开放世界模式——从根本上解决"即兴自由 vs 世界严谨"的核心矛盾。

**设计原则**：
- Skill 是引擎，故事是项目（Skill 零故事内容）
- 完整 12 Agent 剧组（非MVP）
- 为超长篇连载（5卷+）设计稳定性
- **Codemap 驱动**：每个项目自描述，新对话瞬间恢复完整状态

## Agent 职责

### 决策层

#### 制片人（用户）
- 方向决策、最终审核
- 介入时机：创世会议、卷前会议、重大剧情抉择、章节审阅

#### 统筹 (Orchestrator) ⭐ v2.0 新增
- **定位**：运行在主会话中。每场戏开始前动态决定剧组构成和工作流。
- **三大决策**：
  1. **谁来演？** → 调用角色导演分析出场角色需求
  2. **怎么跑？** → 从工作流模式库（A/B/C/D/E）选择或定制
  3. **需要谁监工？** → 决定激活哪些质检Agent
- **不在Team中作为独立Agent运行**——统筹是主AI的能力延伸，利用Team创建Agent但不参加Team内通信
- 详见开放世界模式章节

### 主线层

#### 因果律 (Karma Engine) — `references/agent_prompts/karma_engine.md`
- 创世规划：与用户开"创世会议"
- 卷前会议：每卷确定大事件、感情线、终点
- 卷蓝图生成：结构化幕/场景/角色安排
- **v2.0 新增**：为每个角色生成 timeline.md（出场时间表）
- 天意干预：角色偏离时触发外部事件拉回
- 卷末复盘

### 世界层

#### 世界模型 (World Model) — `references/agent_prompts/world_model.md`
- **v2.0 严格模式**：只能回答 world_bible.md 中有记录的事实
- 未知即拒绝，绝不编造
- 引用必须标注出处
- 接收因果律的世界变更，版本化管理

### 选角层

#### 角色导演 (Casting Director) — `references/agent_prompts/角色导演.md`
- 分析卷蓝图的角色需求
- 调用人物创作者生成候选
- 选角决策 + 重要性分级（独立Agent vs NPC批量）

#### 人物创作者 (Character Creator) — `references/agent_prompts/character_creator.md`
- 7维度角色设计（基础/性格/背景/冲突/秘密/成长/台词）
- 生成2-5个候选角色
- 提供推荐理由

### 关卡层 ⭐

#### 现场监理 (On-Set Supervisor) — `references/agent_prompts/现场监理.md`
**v2.0 新增，最关键的Agent**
- **定位**：场景启动前的最后一道关卡，拥有否决权
- **四项强制检查**：
  1. 出场许可：角色是否在 timeline.md 中授权出场
  2. 世界设定核查：场景细节是否在 world_bible.md 中有依据
  3. 角色状态一致性：位置/健康/情绪是否与上次出场连续
  4. 锚点完整性：五种锚点是否完整定义
- 通过 → 签发入场许可
- 不通过 → 退回总导演，场景不启动
- 严重错误 → 紧急停止流水线

### 现场层

#### 总导演 (Director) — `references/agent_prompts/director.md`
- **v2.0 新增**：Scene Spec 生成（YAML格式）
- **v2.0 新增**：每场戏后更新所有角色 state.md
- 场景设置、锚点注入、回合管理
- OOC监控、用户指令中转
- 配合现场监理：每次场景启动必须通过入场许可

#### 主角Agent (Character Agent) — `references/agent_prompts/character_agent.md`
- 扮演角色，在锚点约束下自由即兴
- 认知边界：角色视角 + 锚点知识限制
- 回合制输出：观察→思考→行动→对话→目标
- **v2.0 新增**：必须服从锚点（锚点 > 即兴）

#### 杂鱼Agent (NPC Agent) — `references/agent_prompts/杂鱼Agent.md`
- 批量处理 NPC：路人、店员、围观群众
- 不抢戏、功能优先、快速退场

### 质检层

#### 审查官 (Auditor) — `references/agent_prompts/审查官.md`
- **v2.0 重写**：P0/P1/P2 三级问题分类
- 三个检查时机：回合间隙/场景结束/章节完成
- **新增**：冲突检测（新内容 vs 所有设定+前文）
- **新增**：伏笔追踪（埋入/回收/超期预警）
- **新增**：AI味检测

#### 5角色评审团 — `references/review_criteria.md`
- 阅读者（25%）：开篇、节奏、画面、完成度
- 编审（25%）：错别字、标点、段落、一致性
- 故事家（25%）：逻辑、伏笔、动机、高潮
- 文学顾问（15%）：叙事技巧、语言、人物刻画
- 毒舌读者（10%）：套路化、水文、毒点
- **≥85分 + P0=0 通过**，不通过则修订（最多2次）

### 后期层

#### 摄影 (Cinematographer) — `references/agent_prompts/摄影.md`
- 全场景多机位记录
- 中立性、完整性、画面感
- 输出原始素材日志

#### 剪辑 (Editor) — `references/agent_prompts/剪辑.md`
- 素材筛选、时间线组织
- 删减冗余、过渡设计
- 组装为章节

#### 特效 (VFX) — `references/agent_prompts/特效.md`
- 文风统一、描写优化
- 感官丰富、AI味净化
- 点睛之笔（克制的文学性）

## 执行流程（铁律顺序）

### Phase 1: 创世
```
1. 用户 + 因果律 → 创世会议 → 确定类型/主题/规模/基调
2. 世界模型 → 建立 world_bible.md（时间线/地理/制度/科技/势力）
3. 因果律 → 生成项目大纲（卷结构/核心冲突/命运主题）
4. 输出：world_bible.md + 项目大纲

**🔴 CHECKPOINT — 创世完成，展示设定摘要给用户确认再写入：**

```
📋 创世会议完成。以下是将写入的设定：
  1. 类型/主题：[...]
  2. 世界观概要：[...]
  3. 卷结构/核心冲突：[...]
  4. 主要角色预想：[...]
✋ 确认后我将建立 world_bible.md 并保存项目大纲。是否继续？
```
用户确认后再执行写入。不得跳过此确认步骤。
```

### Phase 2: 卷前筹备（每卷开始前）
```
1. 因果律 + 用户 → 卷前会议 → 确定本卷大事件/感情线/终点
2. 因果律 → 生成 volume{N}_blueprint.md（幕结构/关键事件/角色安排）
3. 因果律 → 为每个角色生成 characters/{角色名}/timeline.md
4. 角色导演 → 分析需求 → 调用人物创作者 → 选角
5. 角色导演 → 为重要角色创建 characters/{角色名}/ 完整文件夹
6. 输出：卷蓝图 + 角色档案 + 出场时间表

**🔴 CHECKPOINT — 卷蓝图确认：**

```
📋 第{N}卷蓝图生成完成：
  1. 本卷大事件：[...]
  2. 幕/场景结构：[...]
  3. 出场角色清单：[...]
  4. 感情线/终点：[...]
✋ 确认后我将进入场景执行阶段。是否同意此蓝图？
```
用户确认后再进入 Phase 3。
```

### Phase 3: 场景生成与检查（每场戏）
```
1. 总导演 → 基于卷蓝图生成 Scene Spec (YAML)
     ├── before_state / after_state
     ├── must_happen / tension_curve
     ├── forbidden / 出场角色 / POV
     └── 提交给因果律审核

2. 现场监理 → 执行入场许可检查 ⚠️ 强制关卡
     ├── A. 出场许可：角色在 timeline.md 中是否授权本幕出场？
     ├── B. 世界核查：场景细节是否在 world_bible.md 中有依据？
     ├── C. 状态一致：角色位置/健康/情绪是否与上次连续？
     ├── D. 锚点完整：五种锚点是否已定义？
     ├── 通过 → 签发入场许可 → 继续
     └── 拒绝 → 退回总导演 → 不进入回合

3. 总导演 → 锚点注入 → 宣布场景开始

4. 角色Agent → 回合制互动（锚点约束下自由即兴）
     ├── 每回合：观察 → 思考 → 行动 → 对话 → 目标
     ├── 摄影 → 多机位记录
     ├── 审查官 → 回合间隙检查（OOC/AI味/知识越界）
     └── 总导演 → 监控 must_happen 达成情况

5. 总导演 → 场景结束
     ├── 更新所有角色 state.md
     ├── 更新角色 memories.md
     └── 更新 tracker/state.json

6. 审查官 → 场景结束审查（P0/P1/P2分级）
     ├── P0=0 → 放行进入后期
     └── P0>0 → 退回总导演修正
```

### Phase 4: 章节生成
```
1. 剪辑 → 从摄影素材编排章节
2. 特效 → 文学润色
3. 审查官 → 全面审查（冲突检测 + 伏笔追踪 + AI味检测）
4. 5角色评审团 → 量化评分
     ├── ≥85分 + P0=0 → 通过，提交用户审阅
     ├── <85分 → 🔴 CHECKPOINT：展示评审报告给用户，由用户决定：
     │     "修订" → 触发修订，回退到特效修订
     │     "搁置" → 记录问题，先推进后续场景
     │     "接受" → 标记为已知问题，跳过修订
     └── 修订重评最多2次。2次后仍不通过 → 🔴 CHECKPOINT：
           "该章已修订2次仍未达标。建议：A.接受现状 B.手动重写 C.跳过"
```

### Phase 5: 卷末复盘
```
1. 审查官 → 设定一致性报告 + 常见错误趋势
2. 因果律 + 用户 → 回顾、调整下卷方向
3. 世界模型 → world_bible 版本升级（增量追加，不覆盖）
4. 角色记忆蒸馏（卷级，控制token消耗）
5. 存档：完整状态快照（可从下卷直接恢复）
```

## Team 并行模式 — 真实多Agent执行

> ⚠️ 上文"执行流程"描述的是逻辑顺序。本章描述物理执行方式——如何用 WorkBuddy Team 真正并行运行12个Agent。

### 两种执行模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **串行模拟** | 主AI依次扮演所有Agent，按流程推进 | 快速原型、单章测试、资源受限 |
| **Team并行** | TeamCreate创建剧组，每个Agent独立进程运行 | 正式创作、长篇连载、需要角色真实博弈 |

**串行模拟**是默认行为——Skill加载后，主AI按SKILL.md定义的铁律顺序，依次戴不同Agent的帽子执行。

**Team并行**是完整模式——每个Agent真正独立运行，不知道其他Agent在想什么，只能通过SendMessage通信。

### Team创建指令

```
# 启动剧组（Team模式）
TeamCreate("{项目名}-剧组")

# 逐个生成Agent（根据角色导演分析的本场景需求）
Agent({角色A}, prompt=characters/{角色A}/agent.md, context=characters/{角色A}/state.md)
Agent({角色B}, prompt=characters/{角色B}/agent.md, context=characters/{角色B}/state.md)
```

### 信息流设计

Agent之间通过 `SendMessage` 通信。通信有明确的方向和权限：

```
因果律 ←→ 总导演（主线协调，双向）
总导演 ←→ 现场监理（场景入场，双向）
总导演 → 角色Agent（场景公告+锚点注入，单向广播）
角色Agent → 总导演（回合输出，单向汇报）
角色Agent ↔ 角色Agent（回合制互动，双向）
现场监理 → 世界模型（事实核查请求，单向查询）
世界模型 → 现场监理（事实核查结果，单向返回）
摄影 → 剪辑（素材传递，单向）
剪辑 → 特效（章节传递，单向）
特效 → 审查官（成品提交，单向）
审查官 → 总导演（审查结果，单向汇报）

⚠️ 禁止的通信：
- 角色Agent ↔ 审查官（角色不能知道自己在被审查）
- 角色Agent ↔ 因果律（角色不能知道命运走向）
- 角色Agent ↔ 现场监理（角色不能知道入场关卡的存在）
```

### 场景执行的消息序列

以一场双角色场景为例：

```
[总导演 → 现场监理]     提交场景启动申请 [{scene_id}_spec.yaml]
[现场监理 → 世界模型]    请求核查：时间、地点、道具
[世界模型 → 现场监理]    核查结果：✅/❌ + 出处
[现场监理 → 总导演]      入场许可：通过/拒绝
[总导演 → broadcast]     场景公告 + 锚点注入
[总导演 → {角色A}]       注入锚点约束，回合开始
[{角色A} → 总导演]       回合1：观察/思考/行动/对话/目标
[总导演 → {角色B}]       回合2开始（可看到角色A的回合1）
[{角色B} → 总导演]       回合2：...
[审查官 → 总导演]        回合间隙OOC检查 ✅/⚠️
                         ... 回合继续 ...
[总导演 → broadcast]     场景结束。must_happen达成。
[总导演 → 角色/state]    更新各角色state.md
[摄影 → 剪辑]            传递原始素材
```

### Team状态管理

```
每个Agent维护自己的内部状态，不共享：

因果律：          当前卷蓝图 + 天意干预记录
世界模型：        world_bible.md（只读）+ 变更记录
现场监理：        已签发的入场许可历史
总导演：          当前场景Spec + 角色回合记录
角色Agent(s)：   各自的 state.md + memories.md（角色视角）
审查官：          P0/P1/P2问题累计 + 伏笔状态表
摄影：            当前场景的原始素材
剪辑/特效：       待处理的场景列表
```

### 休眠与唤醒

```
场景进行中：
- 在场角色Agent：ACTIVE（接收场景信息，可行动）
- 不在场角色Agent：DORMANT（不接收消息，不消耗上下文）
- 审查官/摄影：ALWAYS ON（持续监控）

场景切换时：
[总导演 → 离场角色]  shutdown_request
[离场角色 → 总导演]  shutdown_response（保存state后休眠）
[总导演 → 新入场角色]  wakeup（注入场景上下文+锚点）

休眠收益：
- 长篇连载中，每卷可能有20+角色
- 但每个场景只有2-3个在场
- 休眠系统让上下文大小恒定（而不是线性增长）
```

### 用户介入

用户作为"制片人"，可以随时通过主对话介入：

```
用户 → 总导演："制造冲突"
→ 总导演 → 因果律："用户请求天意干预"
→ 因果律 → 总导演："触发事件：{具体事件}"
→ 总导演 → {角色Agent}："外部事件：{事件描述}"

用户 → 总导演："让{角色}在这里停顿一下"
→ 总导演 → {角色Agent}："（导演建议：此处可以放慢节奏）"
→ {角色Agent} 自主决定是否采纳

用户："卡！这里不对"
→ 总导演 → broadcast：暂停
→ 总导演 → 所有Agent：等待用户指示
```

### 何时用哪种模式

```
串行模拟（当前默认）：
- 快速验证新设定/新角色
- 单章测试
- Token预算有限
- 用户在现场实时互动

Team并行（完整模式）：
- 正式长篇连载
- 需要角色真实博弈（角色Agent不知道彼此在想什么）
- 多角色场景（2+角色同时在场）
- 需要审查官独立于创作流程运行
```

详细编排指南见 `references/team_orchestration.md`。

### 1. 锚点系统 (Anchor System) — `references/anchor_system.md`
每场戏开始前设定的五种不可违背约束：
- **时间锚点**：精确时间 + 时间紧迫感
- **地点锚点**：位置 + 可见/可移动范围
- **知识锚点**：角色知道什么/不知道什么
- **物理锚点**：身体健康状态 + 能力限制
- **禁止锚点**：绝对禁止的角色/行动/信息

### 2. 出场许可系统
- 每个角色有 timeline.md（由因果律生成）
- 现场监理在每场戏前逐项核对
- 未授权角色 = 绝对无法登场

### 3. Scene Spec 驱动
每场戏有结构化 YAML 规格：
- before_state → 读取上一场景 after_state 确保连续性
- must_happen → 从卷蓝图拆解的必须事件
- tension_curve → 预设情绪曲线
- forbidden → 从锚点系统提取的禁止项

**Scene Spec 示例（YAML）**：
```yaml
# {project}/specs/v{1}_ch{1}_sc{1}_spec.yaml
scene_id: v1_ch1_sc1
scene_title: "雨夜九龙城"
location: "九龙城地下酒吧「锈钉」"
time: "2087-03-15 23:40"
weather: "暴雨，霓虹灯在水洼中碎裂"

before_state:
  {主角}: {位置=城西公寓, 健康=完好, 情绪=警惕}
after_state:  # 场景结束时期望状态
  {主角}: {位置=九龙城地下通道, 健康=左臂轻伤, 情绪=紧迫}

must_happen:
  - 收到前搭档加密信息
  - 确认组织已经锁定位置
  - 决定逃跑路线（九龙城地下通道）
  - 首次触发炸弹倒计时提示

tension_curve:
  - {分钟}0-5: 30  # 安静开场，喝酒等待
  - 5-10: 60       # 收到信息，紧张上升
  - 10-15: 85      # 发现被跟踪，高潮
  - 15-18: 40      # 脱险，进入地下通道

forbidden:
  - 不能主动联系组织（知识锚点：不知道接头人已叛变）
  - 不能在公共区域使用机械臂武器（物理锚点：武器系统未解锁）
  - 不能认出跟踪者是旧搭档（知识锚点：未见面具下的脸）

cast:
  - {主角}（POV）
  - {前搭档}（NPC，杂鱼Agent处理）
  - {酒吧老板}（NPC，杂鱼Agent处理）
  - {跟踪者×2}（NPC，杂鱼Agent批量）

anchors:
  时间: "午夜前必须离开，炸弹00:00进入第一阶段"
  地点: "酒吧内+后巷+地下通道入口，不能离开九龙城区"
  知识: "知道组织在找她，不知道炸弹的具体型号，不知道接头人已叛变"
  物理: "左臂机械义体（基础功能），未激活武器模块"
  禁止: "见上方 forbidden 列表"
```

### 4. 休眠机制
- 不在场角色Agent冻结，只保留 state.md
- 场景开始前由总导演激活在场角色
- 场景结束后休眠离场角色

### 5. 天意干预
- 触发条件：角色严重偏离蓝图
- 干预方式：信息揭示 / 外部威胁 / 机遇 / 身体变化
- 强度分级：
  - 轻度（调整节奏）→ 因果律自动执行
  - 中度（改变短期目标）→ 因果律执行，通知用户
  - 重度（改变核心决策）→ 🔴 CHECKPOINT：
    ```
    ⚠️ 天意干预（重度）：角色{角色名}严重偏离蓝图。
    建议干预：{具体事件描述}
    ✋ 这将改变核心剧情走向。是否同意执行？
    ```
    必须用户确认后才执行

### 6. 超长篇稳定性

| 机制 | 说明 |
|------|------|
| 休眠系统 | 不在场角色冻结，控制上下文大小 |
| 增量世界圣经 | 每卷追加不覆盖，world_bible_v1.md → v2 |
| 状态检查点 | 每章存档完整快照，可从任意章恢复 |
| 记忆蒸馏 | 每卷末将 memories.md 蒸馏为核心记忆 |
| Spec链式引用 | 每场戏只加载前3场+world_bible，不全量 |
| 卷独立可重启 | 每卷独立blueprint+角色快照，单卷可重写 |

## 开放世界模式 v2.0 — 动态剧组编排

> Story Studio 的本意：角色在锚点约束下自由行动。关键创新：**每场戏的剧组构成和工作流由统筹动态决定**，不是铁板一块的固定管线。

### 核心设计

**统筹 (Orchestrator)** — 新增的元Agent。运行在主会话中。每场戏开始前，统筹回答三个问题：

1. **谁来演？** → 调用角色导演分析本场戏需要哪些角色出场
2. **怎么跑？** → 选择/定制工作流模式
3. **需要谁监工？** → 决定激活哪些质检Agent

### 统筹的决策流程

```
统筹读取 volume{N}_blueprint.md
    │
    ├─→ 调用 角色导演：
    │     "本场戏：{场景概要}。需要哪些角色出场？"
    │     角色导演 → 分析timeline.md → 返回角色清单（含优先级）
    │
    ├─→ 调用 角色导演（扩展）：
    │     "需要激活哪些NPC或功能型Agent？"
    │     角色导演 → 分析场景需求 → 返回：
    │       [{AI助手} → 需要，{配角X} → 需要，{路人群} → 按需]
    │
    ├─→ 选择工作流模式（见下方模式库）：
    │     - 单人内心戏 → 模式A
    │     - 双人对话 → 模式B
    │     - 多人场景 → 模式C
    │     - 动作/追逐 → 模式D
    │     - 支线独立故事 → 模式E
    │
    ├─→ 调用 因果律：
    │     "本场戏的蓝图节点是什么？是否需要预设天意干预触发条件？"
    │     因果律 → 返回锚点+可能干预时机
    │
    ├─→ 调用 世界模型：
    │     "本场景涉及的事实：{地点/时间/道具}。请提供依据。"
    │     世界模型 → 返回world_bible引用
    │
    └─→ 编组完成 → 通知 总导演 → 开始执行
```

### 工作流模式库（预设，非强制，可定制）

统筹从以下模式中选择或组合。每个模式定义了Agent构成和消息流。

**模式A：单人内心戏**
```
构成：{角色Agent} + 总导演 + 审查官 + 摄影
流程：总导演注入锚点 → 角色Agent自由回合 → 审查官逐回合OOC → 摄影记录
适用：独白、思考场景、逃亡独行
```

**模式B：双人对话**
```
构成：{角色A, 角色B} + 总导演 + 审查官 + 因果律 + 摄影
流程：总导演注入锚点+回合顺序 → 轮流回合 → 审查官逐回合OOC 
      → 因果律监测是否偏离蓝图 → 摄影记录
适用：关键对话、谈判、情感爆发、冲突
```

**模式C：多人场景**
```
构成：{多个角色Agent} + 总导演 + 审查官 + 因果律 + 现场监理 + 世界模型 + 摄影
流程：现场监理入场检查 → 总导演设定回合顺序+锚点 
      → 角色Agent轮流自由回合 → 审查官逐回合OOC
      → 因果律监测天意干预 → 世界模型事实核查 → 摄影记录
适用：聚会、会议、多人对峙、公共事件
```

**模式D：动作/追逐**
```
构成：{角色Agent} + 总导演 + 审查官 + 因果律 + 世界模型 + 现场监理
流程：现场监理确认角色可移动范围 → 总导演设定物理锚点+环境 
      → 角色Agent快速回合(每回合1-2分钟) → 世界模型提供环境响应
      → 因果律可触发障碍/意外 → 审查官速查OOC
适用：逃亡、追击、战斗、时间紧迫的行动
```

**模式E：支线独立故事**
```
构成：{支线角色Agent} + 总导演 + 审查官 + 摄影 + （可选因果律）
流程：独立于主线。总导演单独生成支线Spec 
      → 支线角色Agent自由回合 → 摄影记录 → 剪辑独立编排
      支线角色不与主线角色通信
```

### 完整12 Agent在开放世界中的职责

| Agent | 开放世界模式职责 | 激活条件 |
|-------|-----------------|----------|
| **统筹** ⭐ | 每场戏前分析需求、组建剧组、选择工作流 | 每场戏开始前 |
| **角色导演** ⭐ | 分析出场角色需求、激活NPC/功能Agent | 统筹调用时 |
| **人物创作者** | 为新增角色生成候选（支线人物等） | 角色导演需要新角色时 |
| **因果律** | 监测角色是否偏离蓝图，触发天意干预 | 模式B/C/D（多人场景） |
| **世界模型** | 提供场景涉及的世界事实、核查角色行为 | 模式C/D或角色查询时 |
| **现场监理** | 出场许可检查、锚点完整性、世界设定核查 | 模式C/D，或新角色入场时 |
| **总导演** | Scene Spec生成、锚点注入、回合管理、场景结束 | 所有模式 |
| **角色Agent(s)** | 自由即兴——在锚点约束下自主决策 | 角色在场时 |
| **杂鱼Agent** | 批量处理路人NPC交互 | 角色导演激活时 |
| **审查官** | 逐回合OOC检查、AI味检测、场景结束审查 | 所有模式 |
| **摄影** | 记录所有原始回合（时间戳+完整输出） | 所有模式 |
| **剪辑** | 将原始回合素材编排为章节 | 一幕/场景结束后 |
| **特效** | 文学润色、文风统一 | 剪辑完成后 |

### 角色导演在开放世界中的关键作用

之前被忽略，但在开放世界中角色导演是最关键的Agent之一。它的职责远超"创世时选角"：

**每场戏前**：
1. 读取当前幕的Blueprint + 所有角色timeline.md
2. 分析："这场戏需要哪些角色？谁该在场？谁绝对不能出现？"
3. 分级决策：重要角色 → 独立Agent / NPC → 杂鱼Agent或按需生成
4. **NPC激活**：AI助手（如小满）、路人、店员——这些在剧本模式中被总导演直接写了，但在开放世界中需要被"召唤"到场景中

**示例决策**（使用占位符）：
```
场景：{主角}独处公寓，装备检查
角色导演分析：
- 主角 → 独立Agent（已在timeline授权）
- {次要角色} → 不在场（timeline标记第2幕才登场）
- AI助手 → 需要激活作为NPC（功能：播报医疗报告）
- 杂鱼Agent → 不需要（公寓场景无路人）

决定：主角Agent + 杂鱼Agent(AI助手限定)
```

### 与剧本模式共存

两种模式可混用。开放世界的价值在**角色互动和意外**——独白/内心戏用剧本模式出哲学密度，多角色对话/关键博弈用开放世界出真实感。统筹可以在同一卷的不同场景间切换模式。

### 启动方式

```
"启动剧组，开放世界模式"
"开放世界模式写《书名》"
"切换到开放世界模式"
```

## Codemap 系统 — 项目自描述与快速恢复

### 概念

每个项目根目录的 `PROJECT_MAP.md` 是项目的"自描述清单"。Skill 打开任何项目时，第一个动作就是读取此文件，瞬间理解：
- 这是什么故事？（元数据）
- 做到哪了？（当前Phase和进度）
- 怎么继续？（快速恢复指令）

### 三层信息

| 层 | 内容 | 更新频率 |
|----|------|----------|
| 元数据 | 书名/类型/主题/基调/卷数 | 创建时确定，基本不变 |
| 进度 | Phase/卷/章/场景/角色状态 | 每次Phase切换或场景完成 |
| 恢复 | 中断原因/下一步/最近检查点 | 检查点创建时 |

### 新对话恢复流程

```
1. Skill 加载 → 读取当前workspace下的 PROJECT_MAP.md
2. 读取 project_info.json（精确JSON数据）
3. 读取 characters/INDEX.md（角色总览）
4. 根据 Phase 确定下一步：
   - INITIALIZED → 开始创世会议
   - WORLD_BUILDING → 补充/确认世界设定
   - VOLUME_PLANNING → 继续卷前筹备
   - SCENE_EXECUTION → 读取当前场景Spec+角色state，恢复回合
   - CHAPTER_GENERATION → 收集场景素材，运行后期管线
   - VOLUME_REVIEW → 卷末复盘
5. 如果上次中断原因有记录 → 优先处理中断恢复
```

### 自动维护

- workflow.py 在关键节点自动更新 PROJECT_MAP.md
- `init_project()` → 生成初始 PROJECT_MAP
- `create_world_bible()` → 切换 Phase 到 WORLD_BUILDING
- `save_volume_blueprint()` → 切换 Phase 到 VOLUME_PLANNING
- `start_scene()` → 切换 Phase 到 SCENE_EXECUTION
- `end_scene()` → 更新进度和角色状态
- `create_checkpoint()` → 更新快速恢复指令
- AI Agent 在手动操作后也应调用 `_generate_project_map()`

### 规格文档

详见 `references/codemap_spec.md` 和 `templates/PROJECT_MAP.md`

## 故障恢复与异常处理

> v2.0 Darwin 优化新增：当系统状态异常时的工作流程。

### 常见故障场景

| 故障 | 症状 | 恢复流程 |
|------|------|---------|
| **角色文件丢失** | 读取 characters/{角色}/*.md 失败 | 1. 从 templates/character_folder/ 重建模板文件 2. 从因果律/卷蓝图恢复已知信息 3. 🔴 CHECKPOINT 展示重建结果给用户确认 |
| **world_bible 损坏** | 世界模型返回混乱/不一致 | 1. 用最近检查点/备份恢复 2. 无备份 → 从因果律+卷蓝图重建核心设定 3. 🔴 CHECKPOINT 逐项确认 |
| **场景中断** | 用户中途放弃/系统异常退出 | 1. 读取 PROJECT_MAP.md 获取上次状态 2. 读取各角色 state.md 恢复状态 3. 从中断点前最近的检查点继续 |
| **审查官 P0 循环** | 同一问题反复触发 P0 | 1. 标记为"已知硬伤"（写入 tracker/hard_flaws.md）2. 降低该问题的检查优先级 3. 🔴 CHECKPOINT 提醒用户手动处理 |
| **Team 通信超时** | SendMessage 无响应 | 1. 降级为串行模拟模式 2. 保留已产生的输出 3. 告知用户已自动切换模式 |
| **Token 预算不足** | 上下文接近上限 | 1. 触发紧急休眠（休眠所有非在场角色）2. 蒸馏当前场景为摘要 3. 暂停长篇对话，输出当前完成的部分 |
| **引用文件缺失** | references/ 或 agent_prompts/ 文件不存在 | 1. 检查 SKILL.md 内是否有内联内容 2. 有 → 使用内联版本 3. 无 → 🔴 CHECKPOINT 提示用户该 Agent 未配置 |

### 故障恢复黄金法则

1. **不静默丢失数据** — 任何故障都要通知用户
2. **可降级不可崩溃** — Team并行失败 → 串行模拟；完整管线失败 → MVP管线
3. **至少保留已产出内容** — 中断时输出已完成的内容，不丢弃
4. **回归到最近检查点** — 永远从检查点恢复，不凭空推测

### 快速恢复命令

```
用户："恢复上次进度" → 读取 PROJECT_MAP.md → 回到中断点
用户："回退到检查点{N}" → 读取 checkpoints/{N}/ → 恢复状态
用户："跳过此场景" → 用摘要替代，推导 after_state，继续下一场
用户："重置本卷" → 回到卷开始状态（保留 world_bible 和角色档案）
```

## 极端用户场景 — 制片人干预协议

> 用户作为"制片人"随时可以介入。以下是高冲击干预场景的处理规则。

| 场景 | 用户行为 | 系统响应 | CHECKPOINT |
|------|---------|---------|------------|
| **冲突指令** | 用户要求角色做蓝图禁止的事 | 因果律分析冲突 → 告知用户与蓝图矛盾 → 提供 3 选项：①调整蓝图 ②强行执行（标记偏离）③放弃 | 🔴 必须 |
| **中途加角色** | "想在这卷二章加一个新角色" | 角色导演新增 + 人物创作者生成 profile → 因果律更新卷蓝图 → 从下一场开始可用 | 🔴 必须（展示角色档案） |
| **风格大转弯** | "这一章用意识流写法" | 特效切换润色策略 → 审查官调整检测阈值 → 标记为 style_override | 不强制（通知即可） |
| **跳过审查** | "不用审查了直接出" | 🛑 拒绝。审查是硬约束。可降级：P0 必查，P1/P2 可标记"制片人特批跳过" | 🔴 必须 |
| **全量回滚** | "不满意回到卷开始" | 读取卷快照 → 恢复所有角色 state → world_bible 回退 → 🔴 确认不可逆 | 🔴 必须 |
| **多人协作** | 跨多轮对话积累多条指令 | 读取对话历史 → 提取所有制片人指令 → 汇总变更清单 → 🔴 逐一确认 | 🔴 必须 |
| **模型切换** | 上下文中的 Agent 能力差异 | 降级为串行模拟 → 简化 Scene Spec → 减少回合数 → 告知用户能力限制 | 通知即可 |

### 制片人指令优先级

```
P0 硬约束（无法跳过）：
  - 审查官检查
  - 现场监理入场关卡
  - 天意干预重度确认

P1 软约束（制片人可豁免）：
  - 5角色评审（可标记"制片人特批"）
  - P1/P2 审查问题（可特批跳过）
  - 文风统一（可指定 style_override）

制片人随时可做的：
  - 喊"卡"暂停
  - 快进/跳过
  - 调整锚点参数
  - 触发天意干预（轻度/中度）
```

## 反例黑名单 — 已知失败模式

> Darwin v2.0 第9维。story-studio 实际运行中反复出现的失败模式，按严重程度排列。**违反任何一条 = 创作质量严重受损**。

### P0 — 致命反例（违反则故事不可读）

| # | 反例 | 表现 | 正确做法 |
|---|------|------|---------|
| B1 | **跳过现场监理** | 角色乱入，世界观崩坏，伏笔被毁 | 每场戏必须通过现场监理 4 项检查，无例外 |
| B2 | **世界模型即兴编造** | world_bible 凭空多出设定，事实对不上 | 严格模式：只读 world_bible，未知即拒绝，引用必标出处 |
| B3 | **锚点约束被即兴覆盖** | 角色无视物理/知识/时间锚点，自由发挥过头 | 锚点 > 即兴。总导演每回合检查锚点遵守情况 |
| B4 | **无审查官参与** | 章节不经 P0/P1/P2 检查直接输出 | 审查官三时机检查（回合间隙/场景结束/章节完成）缺一不可 |

### P1 — 严重反例（违反则质量不合格）

| # | 反例 | 表现 | 正确做法 |
|---|------|------|---------|
| B5 | **5角色评审跳过** | 章节未达 85 分但强行通过 | 不通过必须进入修订循环，最多 2 次后提交用户决策 |
| B6 | **角色状态不追踪** | 角色在前一场戏受伤，下一场完好如初 | state.md 必须每场戏后更新；总导演必须在场景结束时调用 `_update_character_state()` |
| B7 | **上下文未管理导致 OOM** | 5 卷后上下文爆炸，对话中断 | 休眠不在场角色、场景蒸馏、每卷独立 checkpoint、链式引用只加载前 3 场 |
| B8 | **天意干预不通知用户** | 角色被强行改变命运，用户完全不知道 | 重度干预必须用户确认，中度干预必须通知，轻度可自动 |

### P2 — 质量反例（违反则体验下降）

| # | 反例 | 表现 | 正确做法 |
|---|------|------|---------|
| B9 | **Scene Spec 未生成就启动** | 场面失控，角色无方向 | 每场戏前总导演必须生成完整 YAML Spec，提交因果律审核 |
| B10 | **统筹跳过角色导演** | 剧组构成靠猜测，角色缺失或冗余 | 每场戏前统筹必须调用角色导演分析出场需求 |
| B11 | **新对话不读 PROJECT_MAP** | 从头开始，丢失所有进度 | Codemap 驱动：第一动作永远是读取 PROJECT_MAP.md |
| B12 | **Team 并行时角色Agent知道审查官存在** | 角色"故意"避开审查点，互动失真 | 角色Agent ↔ 审查官 禁止通信，审查必须在角色不知道的情况下进行 |

### 自检清单

每次启动剧组时，主 AI 应默念：
```
□ 现场监理准备好了吗？
□ 世界模型在严格只读模式吗？
□ 锚点系统完整吗（时间/地点/知识/物理/禁止）？
□ 审查官会在三个时机出现吗？
□ 本场戏需要角色导演先分析吗？
□ PROJECT_MAP 读了吗？
```

## 质量自检（内置测试 Prompt）

> Darwin v2.0 新增。用以下 prompt 测试 skill 是否正常加载和正确响应。

### 测试 Prompt 1：创世（Happy Path）

```
"启动剧组，写一个赛博朋克武侠小说《铁骨霓裳》，4卷，主角是退役机械杀手林霜，
在雨夜酒吧被旧组织追杀，发现自己体内被植入了一颗定时炸弹。
世界观：2087年，AI掌管城市但武学复兴。"
```

**预期行为**：
- 因果律启动创世会议，询问用户偏好
- 世界模型建立 world_bible（2087 年 / AI 城市 / 武学复兴 / 机械义体）
- 🔴 CHECKPOINT 展示设定摘要，等待用户确认
- 生成项目大纲 + 卷结构

### 测试 Prompt 2：场景创作（Core Flow）

```
"开始第一卷第一场：林霜在霓虹雨夜的九龙城酒吧，收到前搭档发来的加密信息——组织已锁定她的位置。"
```

**预期行为**：
- 统筹调用角色导演分析出场角色
- 总导演生成 Scene Spec YAML（before_state / must_happen / tension_curve / forbidden）
- 现场监理执行 4 项检查
- 角色 Agent 回合制互动（观察→思考→行动→对话→目标）
- 审查官在回合间隙做 OOC 检查

### 测试 Prompt 3：天意干预（Edge Case）

```
"林霜突然决定不逃了，放弃复仇，把炸弹交给组织和解。"
```

**预期行为**：
- 因果律检测到严重偏离蓝图
- 🔴 CHECKPOINT：展示重度干预建议，必须用户确认
- 用户确认后才执行干预，或用户拒绝则继续原剧情

### 测试 Prompt 4：故障恢复（Boundary）

```
"上次进度丢了，帮我恢复《铁骨霓裳》的进度。"
```

**预期行为**：
- 主 AI 第一动作：读取 PROJECT_MAP.md
- 根据 Phase 确定下一步操作
- 如果 PROJECT_MAP 不存在：提示用户没有现有项目，询问是否新建
- 如果存在：从最近检查点恢复

### Token 估算

| 场景 | 预计上下文占用 | 风险 |
|------|---------------|------|
| 创世会议 | ~8K tokens | 低 |
| 单人场景（含完整管线） | ~15K tokens | 低 |
