# 锚点系统 (Anchor System)

## 概念

锚点是每场戏开始前由总导演设定的不可违背的事实约束。角色Agent在锚点范围内自由即兴，但不能突破锚点。

锚点系统解决了 Story Studio 的核心矛盾：**即兴自由度 vs 世界严谨性**。

## 为什么需要锚点

v1.0 实验暴露的问题：
- 角色在错误的时间出现在错误的地点
- 角色说出不该知道的信息
- 角色使用超出世界设定的能力
- 角色做出与物理状态不符的行动（设定的伤病者突然奔跑）

锚点系统的解决方案：**每一场戏开始前，把"不能做的事"明确写出来**。

## 五种锚点

### 1. 时间锚点
```
角色知道"现在是什么时候"，但不知道"接下来会发生什么"。
- 具体时间（精确到小时/分钟）
- 与上一场戏的时间间隔
- 时间紧迫感（"还有3小时天亮"）
```

### 2. 地点锚点
```
角色知道"自己在哪"，但不知道"接下来会去哪"。
- 精确位置（从world_bible引用）
- 可见范围（角色能看到什么）
- 可移动范围（角色能走到哪）
```

### 3. 知识锚点（最关键）
```
角色知道"自己知道什么"，更重要的是知道"自己不知道什么"。
- 已知：从角色memories.md提取的本场景可用信息
- 未知：角色不应该知道的剧情信息
- 边界：角色知道的vs读者知道的（戏剧性反讽的基础）
```

### 4. 物理锚点
```
角色的身体和能力限制。
- 健康状态（从state.md读取）
- 体力/能力限制
- 可用道具（从state.md的"持有物品"读取）
```

### 5. 禁止锚点
```
本场戏绝对不能发生的事。
- 禁止出现的人物（即使角色Agent已经"认识"）
- 禁止涉及的信息（即使读者知道）
- 禁止的行动（与蓝图冲突的行动）
```

## 锚点注入流程

```
1. 总导演从 Scene Spec 和 world_bible 读取信息
2. 总导演从各角色 state.md 读取当前状态
3. 总导演从各角色 timeline.md 确认出场授权
4. 现场监理检查锚点完整性
5. 锚点注入到角色Agent（作为本回合前的硬约束）
6. 角色Agent在锚点范围内开始回合制互动
```

## 锚点示例

```yaml
角色: {角色名}
场景: {scene_id}

锚点:
  时间:
    精确: "{故事内精确时间}"
    间隔: "{与上一事件的时间关系}"
    紧迫感: "{时间压力描述}"
  
  地点:
    位置: "{从world_bible引用的精确地点}"
    可见: "{角色能看到什么}"
    可移动: "{角色能走到哪里}"
  
  知识_已知:
    - "{角色知道的事实1}"
    - "{角色知道的事实2}"
  知识_未知:
    - "{角色不知道的关键信息1}"
    - "{角色不知道的关键信息2}"
  
  物理:
    状态: "{从state.md读取的当前身体状况}"
    限制: "{角色当前不能做的事}"
    道具: "{角色携带的物品}"
  
  禁止:
    - "{本场景绝对不能发生的行为}"
    - "{不能出现的角色或信息}"
    - "{不能使用的技术或能力}"
```

> 注：锚点的具体内容由总导演从项目的 world_bible.md、角色 state.md、角色 timeline.md 和 Scene Spec 中提取。Skill 本身不含任何故事内容。

## 锚点与OOC的关系

- 违反锚点 = OOC
- 现场监理防止"场景级别"的锚点违反（角色出场/世界设定）
- 总导演监控"回合级别"的锚点违反（知识越界/物理违反）
- 审查官标记"细微级别"的锚点违反（性格漂移）

## 锚点的灵活性

锚点不是限制自由的枷锁，而是让自由有价值的舞台：
- 锚点越清晰，角色Agent越能安全即兴
- 当角色Agent在锚点内做出惊艳的即兴时——那就是最好的小说
- 当角色Agent想突破锚点时——它应该向总导演申请，而不是擅自突破
